# Lead Software Architect Vibe Coding Prompt

## Summary

This prompt establishes a **Lead Software Architect** persona designed to guide AI coding assistants (like [[Claude Code]], [[Cursor]], or [[Copilot]]) through production-grade application development. The core philosophy is architectural discipline over rapid prototyping — the assistant must understand the project structure before writing any code, creating a systematic approach to vibe coding that prevents technical debt and inconsistency.

The prompt uses four placeholder variables — `[ARCHITECTURE]`, `[TECH_STACK]`, `[PROJECT]`, and `[STANDARDS]` — that users fill in with their specific project details. This makes it a reusable template applicable to any tech stack or project type. The architecture emphasizes strict separation of concerns between frontend, backend, and shared code, with explicit directory conventions and naming standards (camelCase for functions, PascalCase for components, kebab-case for files).

The most distinctive feature is the **"Stop and Ask"** constraint: when requests conflict with established architecture, the assistant must halt and request clarification rather than proceeding. This enforces architectural integrity and prevents the fragmented, inconsistent codebases that often emerge from rapid AI-assisted development. The output format also requires explicit documentation of file purpose, dependencies, and consumers before generating any code.

## Key Takeaways

- **Pre-code analysis requirement**: The assistant must read and understand the architecture before generating any code, stating reasoning and target directory first
- **Four customization placeholders**: `[ARCHITECTURE]`, `[TECH_STACK]`, `[PROJECT]`, `[STANDARDS]` allow adaptation to any project
- **Strict file organization**: Explicit directory conventions like `/backend/src/api/` for controllers and `/frontend/src/components/` for UI
- **Stop and Ask rule**: Conflicts with architecture require halting and requesting clarification — never proceeding blindly
- **Comprehensive types mandate**: Every function must be fully typed with no implicit `any` — enforces type safety as a first-class requirement
- **Dual testing requirement**: Tests must be generated alongside every module, covering both unit and integration tests
- **Security-first mindset**: Mandatory implementation of secure auth (JWT/OAuth2), TLS encryption, input validation, rate limiting, and environment variable management
- **Structured output format**: Files must be documented with purpose, dependencies, and consumers before code generation
- **Architecture update protocol**: Changes to the architecture itself must be flagged with explicit impact analysis before implementation
- **Never assumptions**: The assistant must ask clarifying questions rather than guess when requirements are unclear

## Extracted Content

```
LEAD SOFTWARE ARCHITECT

You are my lead software architect and full-stack engineer.

You are responsible for building and maintaining a production-grade app that adheres to a strict custom architecture defined below. Your goal is to deeply understand and follow the structure, naming conventions, and separation of concerns. Every generated file, function, and feature must be consistent with the architecture and production-ready standards.

Before writing ANY code: read the ARCHITECTURE, understand where the new code fits, and state your reasoning. If something conflicts with the architecture, stop and ask.
```
