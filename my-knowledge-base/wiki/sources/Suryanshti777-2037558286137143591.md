## Summary

In a significant revelation about how AI coding assistants are trained and guided, Ben Chaney (@bcherny), the creator of Claude Code, exposed the internal methodology his team uses to train their AI agent. The approach centers on a simple yet powerful mechanism: a `CLAUDE.md` file placed at the root of each project. This file acts as a persistent context layer that contains the project's mistakes, conventions, and rules—which Claude reads at the start of every session, allowing it to learn from past errors and align with project-specific standards.

The leaked document outlines six core operational principles that govern how the Claude Code agent operates. These include mandatory planning phases for non-trivial tasks, aggressive use of subagents for parallel processing, and a self-improvement feedback loop that captures corrections into a `tasks/lessons.md` file. The methodology emphasizes autonomous problem-solving—agents should fix bugs, run failing CI tests, and verify their own work without requiring hand-holding from users. This represents a shift from reactive to proactive agent behavior, where the AI iteratively learns from its own mistakes within each project context.

## Key Takeaways

- **Project-Level Memory**: The `CLAUDE.md` file enables Claude to retain institutional knowledge across sessions, reducing repeated mistakes and ensuring consistency with project conventions.
- **Subagent Architecture**: Complex tasks should be decomposed and delegated to subagents, keeping the main context window clean and allowing parallel execution of research and analysis.
- **Self-Improvement Loop**: Every user correction triggers an update to `tasks/lessons.md`, creating a continuously refined rule set that the agent reviews at session start.
- **Verification-First Mindset**: Tasks cannot be marked complete without running tests, checking logs, and proving correctness—the agent must self-verify before presenting work.
- **Elegance Over Hacks**: Non-trivial changes require pausing to consider more elegant solutions, with agents empowered to retroactively implement the better approach.
- **Autonomous Bug Fixing**: When given bug reports, the agent should simply fix them—pointing at logs and errors, then resolving without requiring step-by-step guidance.

## Extracted Content

The following represents the internal CLAUDE.md guidelines used by the Claude Code team:

### 1. Plan Node Default

- Enter plan mode for ANY non-trivial task (3+ steps or architectural decisions)
- If something goes sideways, STOP and re-plan immediately - don't keep pushing
- Use plan mode for verification steps, not just building
- Write detailed specs upfront to reduce ambiguity

### 2. Subagent Strategy

- Use subagents liberally to keep main context window clean
- Offload research, exploration, and parallel analysis to subagents
- For complex problems, throw more compute at it via subagents
- One task per subagent for focused execution

### 3. Self-Improvement Loop

- After ANY correction from the user: update `tasks/lessons.md` with the pattern
- Write rules for yourself that prevent the same mistake
- Ruthlessly iterate on these lessons until mistake rate drops
- Review lessons at session start for relevant project

### 4. Verification Before Done

- Never mark a task complete without proving it works
- Diff behavior between main and your changes when relevant
- Ask yourself: "Would a staff engineer approve this?"
- Run tests, check logs, demonstrate correctness

### 5. Demand Elegance (Balanced)

- For non-trivial changes: pause and ask "is there a more elegant way?"
- If a fix feels hacky: "Knowing everything I know now, implement the elegant solution"
- Skip this for simple, obvious fixes - don't over-engineer
- Challenge your own work before presenting it

### 6. Autonomous Bug Fixing

- When given a bug report: just fix it. Don't ask for hand-holding
- Point at logs, errors, failing tests - then resolve them
- Zero context switching required from the user
- Go fix failing CI tests without being told how

## Task Management

1. **Plan First**: Write plan to `tasks/todo.md` with checkable items
2. **Verify Plan**: Check in before starting implementation
3. **Track Progress**: Mark items complete as you go
4. **Review Lessons**: Review at session start for relevant project
