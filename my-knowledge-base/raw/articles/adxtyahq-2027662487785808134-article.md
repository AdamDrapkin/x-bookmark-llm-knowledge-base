It all feels simple until something crashes in production.
You’ve heard these terms before. You haven’t felt them fail yet.

1. Client–Server Architecture

Client sends request. Server processes and responds.
Everything on the internet runs on this pattern.
It looks simple until traffic spikes.

For eg – during a product launch, our single backend instance hit 100% CPU and started dropping requests.

2. IP Addresses

Every server has a numerical identity.
Requests only reach the right machine if routing is correct.
Wrong IP, wrong destination.

For eg – we deployed a hotfix but frontend was still hitting the old server IP.

3. DNS

Converts domain names into IP addresses.
It’s the internet’s translation layer.
DNS changes aren’t instant.

For eg – after a DNS switch, half users saw new version, half saw old one for 30 minutes.

4. Proxy Server

Acts as a middleman for outgoing or incoming traffic.
Can log, filter, or mask requests.
Often invisible until needed.

For eg – we used a proxy to inspect failing third-party API calls during an outage.

5. Reverse Proxy

Sits in front of backend services and routes traffic.
Clients see one endpoint, multiple services sit behind it.
It hides internal complexity.

For eg – Nginx routed /api to Node and /admin to a separate service.

6. Latency

Delay between sending a request and receiving response.
Distance and network hops matter.
Optimization isn’t just code.

For eg – users outside India experienced slower load times until we added regional infra.

7. HTTP / HTTPS

HTTP sends data in plain text.
HTTPS encrypts communication using SSL/TLS.
Production should never expose plain HTTP.

For eg – an expired SSL certificate once blocked login flow entirely.

8. APIs

Defined contract between frontend and backend.
Clear structure prevents chaos.
Versioning matters more than you think.

For eg – changing response shape without versioning crashed our mobile app.

9. REST

Stateless, resource-based architecture over HTTP.
Easy to understand and scale.
Great for CRUD systems.

For eg – /users/:id/orders endpoint powered both web and mobile consistently.

10. GraphQL

Client requests exactly what it needs.
Reduces over-fetching.
Backend complexity increases.

For eg – dashboard payload size reduced significantly after switching.

11. Databases

Persistent storage layer of your system.
If DB struggles, the entire app feels broken.
It’s often the real bottleneck.

For eg – DB CPU spike increased response time even when servers were idle.

12. SQL Databases

Structured schema with strict consistency.
Great for transactional systems.
Strong relational guarantees.

For eg – used Postgres for payments where data integrity was critical.

13. NoSQL Databases

Flexible schema and horizontal scalability.
Designed for evolving data models.
Trades strict structure for flexibility.

For eg – activity logs schema changed weekly without painful migrations.

14. Vertical Scaling

Upgrade a single machine’s resources.
Quick but temporary solution.
Eventually hits hardware limits.

For eg – doubling RAM fixed memory errors for a few months.

15. Horizontal Scaling

Add more machines to distribute load.
Better long-term strategy.
Requires proper load balancing.

For eg – adding replicas instantly reduced CPU pressure during peak hours.

16. Load Balancer

Distributes incoming traffic across instances.
Prevents uneven load distribution.
Improves reliability.

For eg – one server handled 70% traffic before we fixed routing rules.

17. Indexing

Speeds up database queries.
Prevents full table scans.
Small change, massive impact.

For eg – one missing index turned a millisecond query into seconds.

18. Replication

Primary handles writes. Replicas handle reads.
Improves performance and availability.
Adds synchronization complexity.

For eg – heavy analytics queries moved to read replicas to protect transactions.

19. Sharding

Split database horizontally across multiple servers.
Used when single DB can’t scale further.
Adds routing logic complexity.

For eg – planned sharding user data by region once projections crossed threshold.

20. Vertical Partitioning

Split table columns based on usage frequency.
Keeps hot data lightweight.
Improves query efficiency.

For eg – moved rarely accessed metadata to a secondary table.

21. Caching

Store frequently requested data in memory.
Reduces database pressure.
Critical for high-traffic endpoints.

For eg – Redis cache cut API latency for popular endpoint nearly in half.

22. Denormalization

Combine related data to reduce joins.
Improves read performance.
Adds duplication trade-offs.

For eg – embedded profile info in feed queries to reduce DB joins.

23. CAP Theorem

In distributed systems, you can’t have consistency, availability, and partition tolerance together.
You must choose.
Trade-offs are unavoidable.

For eg – during a network split, we chose availability over strict consistency.

24. Blob Storage

Store large files outside relational DB.
Optimized for unstructured data.
Keeps database lean.

For eg – moved image uploads to object storage and reduced DB size growth.

25. CDN

Distributes static assets globally.
Serves content from closest edge location.
Reduces latency significantly.

For eg – landing page load time improved for users outside primary region.

26. WebSockets

Persistent two-way communication channel.
Used for real-time features.
Eliminates repeated polling.

For eg – live status tracking replaced 5-second polling requests.

27. Webhooks

Event-driven communication between systems.
Trigger actions instantly on events.
More efficient than constant checks.

For eg – payment confirmation handled instantly through webhook callbacks.

28. Microservices

Split large app into smaller independent services.
Allows independent scaling and deployment.
Increases operational complexity.

For eg – separated auth and billing services for safer deployments.

29. Message Queues

Enable asynchronous processing.
Prevents blocking main request cycle.
Improves scalability under load.

For eg – email notifications processed via queue during peak traffic.

30. Rate Limiting

Restrict number of requests per client.
Protects infrastructure from abuse.
Maintains system stability.

For eg – a scraping bot nearly overloaded our API before limits were enforced.
